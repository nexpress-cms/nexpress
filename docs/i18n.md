> per-document translations, plugin/theme UI string bundles,
> ICU MessageFormat, RTL support, and the admin tooling that
> ties them together. This guide walks an operator from
> "single-locale site" to "fully localized multi-language
> site" and explains every moving piece.

## 1. Two Kinds of i18n

NexPress treats two surfaces as separate problems with
separate APIs:

- **Collection content** (Phase 12.1) — user-authored data
  rows: page bodies, blog posts, discussions. Each locale is
  a separate row in the same table, linked by
  `translation_group_id`. Uses `defineCollection({ i18n: true })`,
  the admin's per-collection translation tabs, and locale-aware
  URL routing.
- **UI strings** (Phase 12.5 + Phase D) — framework / plugin /
  theme chrome: button labels, taglines, error messages,
  dashboard widget titles. Each is a key → string entry in a
  per-locale bundle, called from server code via
  `await t("key", locale, params)`.

The two layers don't talk to each other: a collection-content
translation row never goes through `t()`, and a UI string
isn't part of any collection. Choose the layer based on who
authors the text — a content team translating an article
uses the collection layer, a developer shipping a plugin uses
the UI-string layer.

## 2. Configuring Locales

The site's locale list lives in **two places** that have to
agree: `nexpress.config.ts` (read by the server runtime) and
`apps/web/src/i18n.config.ts` (read by the Next.js proxy, which
can't import the full server config). The reference app keeps a single source
of truth in `i18n.config.ts` and spreads it into `nexpress.config.ts`:

```ts
// apps/web/src/i18n.config.ts
export const i18nConfig = {
  locales: ["en", "ko"] as const,
  defaultLocale: "en" as const,
};

// apps/web/src/nexpress.config.ts
import { i18nConfig } from "./i18n.config";

export default defineConfig({
  // ...
  i18n: {
    locales: [...i18nConfig.locales],
    defaultLocale: i18nConfig.defaultLocale,
  },
});
```

Locale strings are passed straight through to BCP-47
consumers (`<html lang>`, sitemap hreflang, `Intl.*`), so use
canonical codes — `en`, `en-US`, `ko`, `pt-BR`, `ar`, etc.
The first locale becomes the default if `defaultLocale`
isn't set; sites that want different ordering set both.

`defineConfig` validates that `defaultLocale` appears in
`locales` and that every collection with `i18n: true` is
backed by a top-level `i18n` block.

## 3. Localizing Collections

Flip `i18n: true` on any collection that should support
per-locale rows:

```ts
defineCollection({
  slug: "pages",
  i18n: true,
  fields: [
    { type: "text", name: "title", required: true },
    { type: "richText", name: "body" },
  ],
  // ...
});
```

Then run codegen + migrate:

```bash
pnpm db:generate
pnpm db:migrate
```

The codegen adds two columns to the table:

- `locale text` — which locale this row belongs to
- `translation_group_id uuid` — siblings (translations of the
  same logical document) share this id

Slug uniqueness is re-keyed from `(slug)` to `(locale, slug)`,
so `/en/about` and `/ko/about` can coexist as the same logical
page in two locales.

Behavior at the pipeline:

- **Create** — `locale` defaults to the configured
  `defaultLocale` if omitted, validates against the locales
  list, and gets a fresh `translation_group_id` if none was
  supplied.
- **Update** — `locale` and `translation_group_id` are sticky;
  the pipeline rejects attempts to change them after create.
- **Find** — `findDocuments({ locale })` filters i18n
  collections; non-i18n collections silently ignore the param
  (no `locale` column to match).

Non-i18n collections keep working unchanged — flipping
`i18n: true` on a single collection doesn't affect the rest.

## 4. Public-Site Routing

The proxy (implementation in `@nexpress/app/proxy`; surfaced as
`src/proxy.ts` in every site — Next 16 rename of the legacy
`middleware.ts`) parses the URL's first path segment, strips it if
it matches a configured locale, and forwards the resolved locale to
the server runtime via the `x-np-locale` request header:

- `/en/blog/hello` → `x-np-locale: en`, internal path `/blog/hello`
- `/ko/blog/hello` → `x-np-locale: ko`, internal path `/blog/hello`
- `/blog/hello` (no prefix) → `x-np-locale: <defaultLocale>`

The catch-all (`apps/web/src/app/(site)/[[...slug]]/page.tsx`)
strips the same prefix before looking the page up, then falls
through to i18n collections (e.g. `localized-pages`) keyed on
`(locale, slug)`.

Metadata:

- The root layout reads `x-np-locale` and emits
  `<html lang="...">` so screen readers and crawlers see the
  right language.
- `buildPageMetadata({ locale })` forwards the locale to
  Next.js's `Metadata.alternates.languages`, which renders as
  `<link rel="alternate" hreflang="...">` tags. These power
  Google's hreflang index and let translation tools detect
  sibling pages.
- `buildSitemap()` groups i18n-collection rows by
  `translation_group_id` and emits `<xhtml:link>` alternates
  inside each `<url>` entry, so the sitemap is one row per
  logical document with all its locale alternates listed.
- **Sitemap index split** (Phase 12.9): when i18n is configured,
  `/sitemap.xml` returns a `<sitemapindex>` pointing at one
  child sitemap per locale (`/sitemap.xml?locale=en`, etc.).
  Each child holds only that locale's URLs but every entry
  still carries the full `<xhtml:link>` alternates list.
  Static routes (`/`, `/blog`, `/discussions`, `/search`) and
  non-i18n collection rows live in the default-locale child
  only so a URL never appears in two sibling sitemaps. This
  keeps each child file under sitemaps.org's 50K-entry cap
  even on sites with many locales × many collections.

Admin (`/admin/...`) renders without a locale prefix — admin
chrome is operator-facing.

## 5. Creating Translations in Admin

The collection edit form for an i18n-enabled collection grows
a **TranslationTabs** strip across the top: one chip per
configured locale, with the row's own locale highlighted.

- A locale that already has a sibling row → click navigates
  to that row's edit page.
- A locale that doesn't yet have a sibling → click POSTs to
  `/api/admin/collections/{slug}/{id}/translations`, the
  helper copies non-framework columns from the source row
  (title, body, blocks, etc.) into a new row, links via
  the same `translation_group_id`, and lands as **draft** so
  the translator reviews before publishing.

The slug isn't copied — `applySlugField` derives a fresh one
from the title in the new locale, so `(locale, slug)`
uniqueness can't collide.

Core helpers behind the API:

- `findTranslations(collection, docId)` — sibling rows in
  configured locale order
- `createTranslation(collection, sourceDocId, targetLocale, user)`
  — copy + link

## 6. Visitor Language Picker

`@nexpress/theme-default`'s header automatically mounts a
language picker when `getI18nConfig()` reports more than one
configured locale. The picker:

- reads the current path with `usePathname()`
- replaces the locale prefix when one is recognized,
  prepends one when the URL has no known locale
- emits `<Link>` so the SPA router takes over
- marks the active locale with `aria-current="true"` and
  `data-active="true"` for styling

Active locale styling lives in theme-default's `styles.ts`
(`.np-language-picker` + `.np-language-picker-link[data-active]`).

Themes that want a different shape — drop-down, native
language names, flag icons — replace `slots.header` in their
own `defineTheme()` call and implement the picker however
they like. The contract is just "rewrite the URL's first
segment to a different locale code." See
`packages/themes/default/src/components/language-picker.tsx`
for a 50-line reference implementation.

Sprint S — the picker is sibling-aware. The header server
component calls `resolveAvailableLocales(pathname)` from
`@nexpress/next` and forwards the result to the picker via
`availableLocales`. Locales that don't publish a translation
of the current page render as a disabled `<span>` with
`aria-disabled="true"` instead of a `<Link>` so visitors can't
jump to a guaranteed 404. Static / static-list paths (`/`,
`/blog`, `/search`, `/discussions`, `/u/…`, `/members/…`) and
non-i18n routes stay fully enabled because the URL space
exists in every locale.

## 7. UI Strings (Plugins + Themes)

Plugins and themes ship per-locale string bundles in their
manifest:

```ts
// packages/themes/magazine/src/index.ts
defineTheme({
  manifest: { id: "magazine" /* ... */ },
  impl: {
    // ...
    i18n: {
      en: {
        "magazine.tagline": "Stories, essays, and reports",
      },
      ko: {
        "magazine.tagline": "이야기, 에세이, 그리고 리포트",
      },
    },
  },
});
```

```ts
// packages/plugin-reading-time/src/index.ts
definePlugin({
  manifest: { id: "reading-time" /* ... */ },
  i18n: {
    en: { "readingTime.minLabel": "{minutes} min read" },
    ko: { "readingTime.minLabel": "{minutes}분 읽기" },
  },
});
```

The framework merges every registered bundle at boot. Server
components call `t()`:

```ts
import { t } from "@nexpress/core/i18n";
import { headers } from "next/headers";

export async function MagazineHeader() {
  const headerList = await headers();
  const locale = headerList.get("x-np-locale") ?? undefined;
  const tagline = await t("magazine.tagline", locale);
  // ...
}
```

Lookup chain (with admin overrides — see §9):

1. site-scoped admin override for the requested locale
2. requested-locale plugin / theme bundle
3. site-scoped admin override for `defaultLocale`
4. `defaultLocale` plugin / theme bundle
5. the key itself (last-resort identity fallback so missing
   translations are operator-visible without crashing the
   page)

The locale-locality rule: a requested-locale **bundle** wins
over a default-locale **override**. That keeps an English
override from accidentally bleeding into a fully-translated
Korean page.

`t()` is async because the override layer reads from DB on
first access (per-site cache populated lazily). `tSync()` is
the bundle-only resolver for sync contexts.

## 8. ICU MessageFormat

UI string templates use [ICU MessageFormat](https://unicode-org.github.io/icu/userguide/format_parse/messages/),
backed by `intl-messageformat`:

```ts
// Plain interpolation
"greeting": "Hello, {name}!"

// Plural — English
"items.count":
  "{count, plural, =0 {No items} one {1 item} other {# items}}"

// Plural — Korean (CLDR collapses everything to `other`)
"items.count": "{count, plural, =0 {항목 없음} other {항목 #개}}"

// Select
"auth.greeting":
  "{role, select, admin {Welcome, admin} editor {Hi, editor} other {Hi}}"

// Locale-aware number
"stats.views": "{n, number} views"
// en: "12,345 views"  /  de: "12.345 Aufrufe"

// Locale-aware date
"published.at": "Published on {when, date, long}"
```

The locale used for plural rules / number formatting is the
locale the matched template came from — so a default-locale
fallback message gets default-locale plural rules even on a
request for a locale that's missing the key.

Compiled ICU instances are cached by `(locale, template)` so
hot keys parse once per process. Plain strings without `{`
skip the parser entirely.

Tolerance:

- Malformed ICU template → `warn` log + raw template
  rendered. No 500.
- Missing required param → `warn` log + raw template
  rendered. No 500.

Migration note: pre-12.7 NexPress used a custom `{{name}}`
interpolation. ICU uses single braces (`{name}`). Plain
strings don't change.

## 9. Admin String Overrides

Plugins and themes ship a fixed bundle, but typos and copy
tweaks shouldn't require a code change. The admin
**Settings → Strings** tab layers per-site overrides on top of
every bundle:

- `np_string_overrides(site_id, locale, key, value, ...)`
  table; per-site cache lazy-loaded on first `t()` call,
  invalidated on every set/delete.
- `GET /api/admin/i18n/strings` (editor+) lists the merged
  view. `PUT` and `DELETE` are admin + CSRF.
- Admin UI shows each key, its bundled value (faded), the
  current override (if any), and per-locale editors with
  save + revert buttons. An "override" badge marks keys
  that diverge from the bundle.

Multi-process consistency is eventual: each worker has its
own in-memory cache, so an override change in one process
isn't immediately visible in another. In practice the
`revalidatePath` on the relevant routes (or a redeploy)
brings everyone back in sync.

## 10. RTL Support

Locales whose script is RTL (`ar`, `he`, `fa`, `ur`, etc.)
render with `<html dir="rtl">` automatically. The framework
resolves the direction from `Intl.Locale.textInfo` (CLDR-
backed, ECMA-402 stage 3), so sites don't declare per-locale
direction config.

```ts
import { getLocaleDirection } from "@nexpress/core/i18n";

getLocaleDirection("ar"); // "rtl"
getLocaleDirection("ku-Arab"); // "rtl" — script tag wins
getLocaleDirection("ku-Latn"); // "ltr"
getLocaleDirection("en"); // "ltr"
getLocaleDirection("invalid"); // "ltr" (fallback, no throw)
```

With `dir` on `<html>`, every flex container's main axis
mirrors automatically; CSS `text-align: start/end`,
`margin-inline-start/end`, and `padding-inline-start/end`
resolve to the right side without per-locale CSS.

Authoring tip: when writing theme CSS, prefer logical
properties over physical ones:

```css
/* Won't mirror */
.thing {
  margin-left: 1rem;
  text-align: left;
}

/* Mirrors with dir */
.thing {
  margin-inline-start: 1rem;
  text-align: start;
}
```

`@nexpress/theme-default`, `@nexpress/theme-magazine`, and
`@nexpress/theme-portfolio` all use logical properties for
their directional layout, so each renders correctly in both
LTR and RTL out of the box. The audit is locked in by
`apps/web/tests/theme-magazine-portfolio.integration.test.ts`
("example themes use logical (RTL-safe) properties for
directional layout") — that suite rejects new physical-side
rules (`float: left|right`, `margin-left|right`, etc.) so a
future hand-edit can't silently regress.

Admin chrome stays LTR regardless of visitor locale.

## 11. Translation Progress Dashboard

`Settings → Locales` shows two cards:

1. **Configured locales** — read-only list of which locales
   are enabled and which is the default.
2. **Translation progress** — per-collection × per-locale
   matrix. Each row is one i18n-enabled collection; columns
   are configured locales. Each cell shows the row count
   plus a `-N` amber badge for the count of translation
   groups still missing this locale. Complete locales render
   green-tinted.

Backed by `getTranslationProgress()` (one `GROUP BY` per
i18n collection) and `GET /api/admin/i18n/progress`
(editor+). Useful for spot-checking whether a translation
push completed or for prioritizing the next locale.

## 12. Per-Locale Search and Feeds

The cross-collection search (`searchCollections({ locale })`)
folds the locale into the per-collection where clause for
i18n collections; non-i18n collections silently ignore it.
The site search page reads the current locale from the URL
prefix and forwards it, so visitors browsing under `/ko/`
only see Korean hits.

```ts
import { searchCollections } from "@nexpress/core";

const result = await searchCollections({
  q: "tortoise",
  collections: ["localized-pages", "posts"],
  locale: "ko",
});
```

The pluggable search adapter (`setSearchAdapter()`) gets the
locale on its `NpSearchAdapterContext`, so an Algolia /
Meilisearch / OpenSearch integration can implement per-locale
filtering.

Atom feed:

- `buildAtomFeed({ locale })` restricts an i18n collection's
  feed to one locale.
- `renderAtomFeed` embeds `xml:lang` and includes
  `?locale=` in the feed's self-link.
- `/feed.xml?locale=ko` is a public-site URL.

## 12a. Number / Date Formatting Helpers

For values rendered outside an ICU translation template (a row's
`updatedAt`, a comment count, a price), `@nexpress/core` ships
thin wrappers around `Intl.*Format` that resolve the locale from
the i18n config when the caller doesn't pass one explicitly.
Repeat calls reuse a cached formatter instance, since
`new Intl.NumberFormat(...)` is non-trivial.

```ts
import { formatNumber, formatDate, formatRelativeTime } from "@nexpress/core/i18n";

// Locale resolution: explicit arg → getI18nConfig()?.defaultLocale → "en".
formatNumber(1234.5); // "1,234.5"
formatNumber(1234.5, "de-DE"); // "1.234,5"
formatNumber(99.5, "en-US", { style: "currency", currency: "USD" }); // "$99.50"

formatDate(post.updatedAt, "en-US", { dateStyle: "medium" }); // "Apr 29, 2026"
formatDate(post.updatedAt); // resolved locale + runtime defaults

formatRelativeTime(-2, "day", "en-US"); // "2 days ago"
formatRelativeTime(-1, "day", "en-US", { numeric: "auto" }); // "yesterday"
```

Bad inputs are tolerated (no throws): `NaN` / `Infinity` are
returned as their literal string for debuggability; an
unparseable date returns `""`. Sites that need
`formatToParts` or locale-fallback chaining should keep calling
`Intl.*` directly — these helpers are intentionally narrow.

## 13. Translation Interchange (XLIFF + Gettext)

NexPress ships two content-translation adapters over one
`@nexpress/translation` engine:

- `@nexpress/xliff` emits XLIFF 1.2 for SaaS translation tools such as Crowdin,
  Phrase, Smartling, and Lokalise.
- `@nexpress/gettext` emits Gettext PO for open-source and repository-based
  localization workflows.

The shared engine owns collection discovery, translation sibling lookup,
source-text validation, Lexical preservation, block schema lookup, and writes.
Adapters only parse and render their file format, so both formats fail closed
under the same live-document rules. Fresh `create-nexpress` projects include
both CLI shims and their package dependencies.

### Admin workflow

Editors with the `content.publish` capability can use **Settings →
Translations** for bounded, interactive interchange:

1. Choose one i18n-enabled collection, source locale, target locale, and file
   format, then download the catalog.
2. Upload the translated `.xliff`, `.xlf`, or `.po` file and select **Preview
   import**. The report lists document creates, updates, unit counts, and every
   fail-closed skip without writing to the database.
3. Review the report and select **Apply translations**. The confirmation step
   uploads and parses the file again, resolves the current live source rows,
   and repeats every validation before writing. A preview is never reused as a
   stale execution plan.

The Admin endpoint accepts exactly one source/target locale pair and is capped
at 4 MiB, 100 documents, and 2,500 units per request. Use the CLI for larger
catalogs or automation. Both export and import return `Cache-Control: no-store`;
exports are attachment responses with document and unit count headers. New
translations still land as drafts, and existing target siblings are updated in
place under the same shared engine used by the CLI.

### XLIFF CLI

```bash
# Export per (collection, source-locale, target-locale) into a directory.
pnpm xliff export ./xliff/

# Apply a translator's bundle. Targets land as `status="draft"`.
pnpm xliff import ./xliff/posts-en-ko.xliff

# Validate + report without touching the DB.
pnpm xliff import ./xliff/posts-en-ko.xliff --dry-run
```

XLIFF maps one `<file>` to a logical document using
`original="{slug}/{translationGroupId}"`. Each `<trans-unit>` is an atomic
field, protected Lexical value, or schema-declared block prop.

### Gettext CLI

```bash
# Export one PO catalog per collection and locale pair.
pnpm gettext export ./po/

# Apply translated messages; targets land as drafts.
pnpm gettext import ./po/posts-en-ko.po

# Resolve and validate everything without writing.
pnpm gettext import ./po/posts-en-ko.po --dry-run
```

Every PO message carries an opaque framework-owned `msgctxt` that protects the
document route and unit id. Do not edit it. Atomic `msgid`/`msgstr` values stay
plain text. Rich text contains `{NP:G:...}`, `{NP:/G}`, and `{NP:X:...}`
tokens; translate only text between group tokens and keep every token in its
original order. A missing, changed, or reordered token rejects the PO before
any write.

### Shared content contract

- Atomic `text`, `textarea`, and `email` fields use their field name as the
  unit id. Import checks the serialized source against the current live row;
  content changed after export skips that stale unit instead of applying a
  translation to the wrong revision.
- In XLIFF, `richText` fields use
  `restype="x-nexpress-richtext"`: every Lexical text leaf is a
  protected `<g id="n-…" ctype="x-nexpress-lexical">` code and
  block boundaries are `<x ctype="lb"/>` placeholders. Import
  replaces text leaves only, preserving paragraphs, lists, links,
  formatting metadata, and non-text nodes such as images.
- `blocks` fields emit one unit per `propsSchema` leaf declared with
  `translatable: true`. Unit ids protect the collection field, stable block
  instance id, registered block type, and recursive prop path. Nested block
  order may change because ids drive lookup; array `itemSchema` entries are
  positional and must retain their source order while the bundle is in flight.
- Block import validates the live source block id, type, schema path, and text.
  Duplicate ids, unknown or unloaded block types, schema drift, missing target
  paths, and source-tampered units skip only that unit. Operational strings
  declared `translatable: false` never leave the site in either format.
- Rich-text code integrity is strict. Import compares source code
  ids, order, and text against the current source document, and
  requires the target to retain the same ordered codes. Missing,
  reordered, duplicated, or source-tampered codes skip that field
  instead of guessing. Other valid fields in the same file still
  apply.
- Existing rich-text targets pre-fill the next export when their
  Lexical text-leaf structure is compatible with the source.
  Empty target fragments retain the existing translated leaf (or
  the source leaf on first import), so a partial translator export
  never blanks content.
- New target rows land as **draft** so a reviewer can publish
  through the existing Phase 9.7d promote flow. Existing target
  rows are updated in place — re-importing the same bundle
  doesn't duplicate.
- Source rows must be `status="published"` to ship to a
  translator. Drafts stay in-house.
- Empty target text is skipped (the translator has not finished that unit
  yet); it never blanks an existing translation.

API entry points (for sites that wrap this in a custom UI rather
than the CLI):

```ts
import { exportXliff, importXliff } from "@nexpress/xliff";

const bundle = await exportXliff(); // every i18n collection × every target locale
for (const file of bundle.files) {
  // file.name, file.collection, file.sourceLocale, file.targetLocale, file.xml
}

const result = await importXliff({ xml, user });
// result.applied: [{ collection, docId, locale, operation: "create" | "update", unitCount }]
// result.skipped: [{ reason, ... }]
```

```ts
import { exportGettext, importGettext, parseGettext, renderGettext } from "@nexpress/gettext";

const bundle = await exportGettext();
const catalog = parseGettext(bundle.files[0].po);
// Edit unit.target or group text inside unit.targetInline.
const po = renderGettext(catalog);
const result = await importGettext({ po, user, dryRun: true });
```

Programmatic tooling reads `unit.source` and `unit.target` as plain-text
projections. Rich-text units additionally expose `sourceInline` /
`targetInline`; callers must update only `text` inside `group` parts and retain
all code ids and placeholders.

Block units use framework-owned `np:block:` ids. Programmatic tooling may edit
their `target` or `targetInline` content but must treat the unit id as opaque;
changing it invalidates the live block/schema lookup.
