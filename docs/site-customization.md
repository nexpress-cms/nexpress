# Site customisation (Track A)

This guide covers what you can change in a NexPress site created via
`npx create-nexpress` **without cloning the monorepo**. If your goal
is to ship a site (or run an agency, or build a theme/plugin for
sale), you stay here. The only reason to leave Track A is to change
NexPress framework internals ŌĆö see [the framework
contribution guide](../CONTRIBUTING.md) when that comes up.

The "thin wrapper" pattern means every wrapper file is 1ŌĆō11 lines of
`export { ŌĆ” } from "@nexpress/app/ŌĆ”"`. The real implementation lives
in the published `@nexpress/app` npm package ŌĆö when NexPress ships a
fix, you get it with `pnpm install` and a version bump. Nothing in
your scaffold drifts from the framework unless **you** edit a wrapper
on purpose (see "Unwrap a wrapper" below).

## Surfaces you customise (in order of frequency)

### Site identity and env

`src/nexpress.config.ts` and `.env` (or the `pnpm run setup` wizard).

- `site.name`, `site.url`
- `auth.secret` (NP_SECRET)
- `db.connectionString` (DATABASE_URL)
- Storage adapter ŌĆö `NP_STORAGE_ADAPTER=local` (default) or `s3` with
  `NP_S3_BUCKET` / `NP_S3_REGION` / `NP_S3_ENDPOINT`

The `pnpm run setup` wizard writes the `.env`; manual edits work too.

`defineConfig()` is an exact module-evaluation boundary. The active top-level
keys are `site`, `db`, `storage`, `collections`, `themes`, `i18n`, `auth`,
`plugins`, and `jobs`; unknown keys and unknown nested properties fail instead
of being silently retained or ignored. Runtime auth TTL/lockout values are env
settings (`NP_TOKEN_EXPIRATION`, `NP_REFRESH_TOKEN_EXPIRATION`,
`NP_MAX_LOGIN_ATTEMPTS`, `NP_LOCKOUT_DURATION`), not config properties.

The same non-throwing contract is available to tooling through
`npAnalyzeProjectConfig()` and `npValidateProjectConfig()` from
`@nexpress/core`. It also checks that `site.url` is an HTTP(S) origin, storage
URLs are usable, locales are canonical and unique, and plugin ids/dependencies
form one complete acyclic inventory.

Storage base URLs are path prefixes: absolute local paths such as `/media` or
`/`, and HTTP(S) bases such as `https://cdn.example.com/media`, are accepted.
Credentials, query strings, and fragments are rejected because appending an
object key would discard or reinterpret those components.

### Plugins

```bash
pnpm exec nexpress plugin add @scope/plugin-foo
pnpm exec nexpress plugin remove @scope/plugin-foo
```

This rewrites the `// @nexpress:plugins-imports-*` and
`// @nexpress:plugins-list-*` markers in your `nexpress.config.ts`
automatically ŌĆö keep those marker comments in place. After a successful
add or remove, the CLI prints the restart step plus
`pnpm --silent run ops:plugins -- doctor --json` so you can verify
plugin-owned blocks and routes before deploying. If those markers are
missing, the CLI prints the exact import and `plugins` entry to add or
remove by hand.

Authoring a plugin: separate package, scaffolded with
[plugin-quickstart](./plugin-quickstart.md). That guide is the canonical
place for local plugin packages, admin/config surfaces, page routes,
verification, and upgrade checks. This page only covers the site-level
install/remove flow.

### Themes

Built-in packs already wired by the scaffold:
`@nexpress/theme-default`, `@nexpress/theme-magazine`,
`@nexpress/theme-portfolio`, `@nexpress/theme-docs`. Switch the active
one from admin ŌåÆ Appearance.

To install a third-party theme, one command from the project root:

```bash
pnpm exec nexpress theme add @you/theme-cool
pnpm db:generate && pnpm db:migrate
```

This runs `pnpm add`, edits `nexpress.config.ts` via the
`@nexpress:themes-*` marker comments to insert the import and
append the identifier to the `themes:` array, then leaves the
migration step to you (or use `--apply` to chain it). The
framework auto-merges the theme's `manifest.requires.collections`
into your `collections` array at `defineConfig` time, so your
own `src/collections/*.ts` files stay untouched.

To remove a theme, run the CLI before removing the package:

```bash
pnpm exec nexpress theme remove @you/theme-cool --dry-run
pnpm exec nexpress theme remove @you/theme-cool --yes
pnpm db:migrate
pnpm remove @you/theme-cool -w
```

The remove command unregisters the theme before generating the
destructive `DROP COLUMN` migration. If that theme was active, open
`/admin/settings/theme` after deploy and save the fallback theme as active or
activate another registered theme so the persisted `activeTheme` setting is no
longer stale.

To author a new local theme, scaffold it into the reserved theme workspace:

```bash
pnpm exec nexpress create theme cool --workspace
pnpm --filter theme-cool build
pnpm exec nexpress theme add theme-cool --yes
```

See [theme-quickstart.md](./theme-quickstart.md) for the full first-theme path
and [theme-authoring.md](./theme-authoring.md) for the contract reference. The
manual equivalent of `theme add` is the same import + array entry:

```ts
import { coolTheme } from "@you/theme-cool";
import {
  defaultThemes,
  /* ŌĆ” */
} from "@nexpress/app/config-defaults";

export default defineConfig({
  // ŌĆ”
  themes: [...defaultThemes, coolTheme],
});
```

### A new collection

```ts
// src/collections/projects.ts
import { defineCollection } from "@nexpress/core";

export const projectsCollection = defineCollection({
  slug: "projects",
  labels: { singular: "Project", plural: "Projects" },
  admin: { group: "Content", icon: "FolderKanban" },
  fields: [
    { name: "title", type: "text", required: true },
    { name: "summary", type: "textarea" },
    { name: "body", type: "richText" },
  ],
});
```

`defineCollection()` validates the complete definition while this module is
evaluated. Unknown properties, unsafe or framework-reserved field names,
duplicate names (including fields flattened through row/collapsible layout
containers), duplicate select/radio/block/relationship inventories, inverted
min/max bounds, stale `slugField` or Admin column references, non-`media`
upload targets, and malformed nested Admin conditions fail before codegen or
application bootstrap. Declarations the persistence layer cannot representŌĆösuch
as polymorphic relationships, multi-select persistence, or nested child tables
inside stored group/array recordsŌĆöalso fail instead of being silently omitted.

Tooling can inspect the same contract without throwing through
`npAnalyzeCollectionDefinition()` or `npValidateCollectionDefinition()` from
`@nexpress/core`. Their plural counterparts validate a complete collection set,
including duplicate slugs and relationship targets. `defineConfig()` repeats
definition validation before theme requirements merge and validates the complete
resolved set afterward, covering hand-built objects that bypass
`defineCollection()`.

The generated document type for `body` is `NpRichTextContent | null`. Admin,
API writes, themes, and plugins share the versioned contract documented in
[`rich-text.md`](rich-text.md); raw Lexical `{ root: ... }` JSON is rejected.

Register in `nexpress.config.ts`:

```ts
import { defaultCollections } from "@nexpress/app/config-defaults";
import { projectsCollection } from "./collections/projects";

export default defineConfig({
  // ŌĆ”
  collections: [...defaultCollections, projectsCollection],
});
```

Then:

```bash
pnpm db:generate      # writes the new Drizzle schema
pnpm db:migrate       # applies the new tables
pnpm dev              # admin sidebar picks up the new entry
```

### Customising a built-in collection (unwrap)

Built-in `posts` / `pages` / `categories` / `tags` are re-exported
wrappers like:

```ts
// src/collections/posts.ts
export { postsCollection } from "@nexpress/app/collections/posts";
```

To override fields/hooks/access: open the wrapper and replace it with
a full definition. From that point on your `posts` is yours ŌĆö framework
updates to `@nexpress/app/collections/posts` no longer reach your site.
This is the explicit seam between Track A and "I want to fork this
file."

### CSS / Tailwind

`src/app/globals.css` is small:

```css
@import "tailwindcss";
@import "@nexpress/app/styles/globals.css"; /* framework layers, tokens */
@source "../../node_modules/@nexpress/admin/dist/**/*.js";
@source "../../node_modules/@nexpress/app/src/**/*.{ts,tsx}";
@source "../../node_modules/@nexpress/blocks/dist/**/*.js";
@source "../../node_modules/@nexpress/editor/dist/**/*.js";
```

Add your own `@layer` rules below the framework `@import` to layer on
top without forking. To override a framework token, set it in your
own `@layer` after the import.

### Scripts (advanced)

Each `scripts/*.ts` is a one-liner that loads the matching
`@nexpress/app/scripts/<name>`. Override one by replacing the
import with your own logic ŌĆö `pnpm run setup` / `pnpm run doctor` /
`pnpm run worker` continue to invoke whatever the file in your repo
contains.

## Common questions

**Q: I want to add a sidebar widget to the admin. Track A or B?**

Track B ŌĆö admin UI components live in `@nexpress/admin`. The
appropriate path is to fork the repo or open a PR there. Plugin
authors do have a `pageRoutes` surface (see [plugin-admin.md](./plugin-admin.md))
for adding admin pages without touching `@nexpress/admin` itself ŌĆö
check whether that solves your case first.

**Q: I edited a wrapper. How do I get framework updates again later?**

Delete your fork of that file and replace it with the wrapper version
that ships in the current `create-nexpress` scaffold (or copy from
the matching version of `apps/web` in the monorepo). One file at a
time; the rest of your site keeps receiving framework updates as
usual.

**Q: How do I author a brand-new admin shell or admin theme?**

Today, no. The admin shell ships from `@nexpress/admin` and isn't
extensible from a Track A site. Open an issue describing the
use-case ŌĆö if it's common, the right answer is a new contribution
surface, not an unwrap.

**Q: My organisation needs an audit log of every framework version
change. How?**

`@nexpress/*` packages follow the lockstep fixed-versioning policy:
every release of the family bumps every member to the same version.
Subscribe to GitHub releases on `@nexpress/app` (the dependency
graph root for Track A sites) ŌĆö every site receives the same set of
fixes on the same release.

## Going to Track B

If you find yourself wanting to edit a file under `node_modules/@nexpress/...`
to debug something, that's the signal to clone the monorepo and
work there. Set-up steps are in the [root README](../README.md#monorepo).

Once you have a fix, the contribution path is in
[CONTRIBUTING.md](../CONTRIBUTING.md).
