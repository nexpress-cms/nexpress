# NexPress 설계서 리뷰

> 리뷰 대상: `nexpress-core-design.md`, `plugin-system-design.md`, `nexpress.txt` (기획서 v3)
> 리뷰 일시: 2026-04-17
> 상태: 코드 작성 전 (설계 단계)

---

## 전체 평가

코어 CMS 설계는 구현 가능한 수준에 근접해 있으나,
**인증/세션 일관성, 플러그인 생명주기 모델, 라우팅 계약, 백그라운드 작업** 영역에 구현 차단급 이슈가 있다.

**가장 큰 모순**: 코어는 "빌드 타임 코드 생성" 모델인데, 플러그인 시스템은 "런타임 확장 플랫폼"처럼 설계되어 있다. 이 충돌을 코드 작성 전에 해결해야 한다.

**설계 보정 예상 소요**: 약 6-7일 (문서 수정). v1에서 런타임 비신뢰 플러그인까지 포함 시 대폭 증가.

---

## 🔴 구현 차단 (Critical Blocker) — 6건

### CB-1. 인증/세션 모델 내부 모순

**분류**: 모순, 보안

refresh token 엔드포인트가 없다. 로그인 핸들러가 `nx-refresh` 쿠키를 설정하지만,
이를 소비하는 `POST /api/auth/refresh` 라우트가 설계에 없다.
access token이 2시간 후 만료되면 사용자가 재로그인해야 한다.

비밀번호 변경 후 세션 불일치: `invalidateAllSessions()`가 `nx_sessions`의 모든 레코드를 삭제한 뒤,
새 JWT를 발급하지만 새 세션 레코드를 생성하지 않는다.
이 상태에서 `verifyTokenFull()`이 DB를 조회하면 세션이 없어 인증이 실패한다.

미들웨어 vs API 핸들러 인증 분리: Edge 미들웨어는 JWT 서명만 검증하고,
API 핸들러는 DB `tokenVersion`까지 확인한다.
관리자가 세션을 무효화해도 미들웨어는 만료되지 않은 JWT를 계속 통과시킨다 (최대 2시간).

추가 누락 항목:

- refresh token rotation / replay 방어
- CSRF 보호 (cookie 기반 인증에 필수)
- 명확한 "단일 인증 검증 경로" 정의 (미들웨어, RSC, API 핸들러 간)

### CB-2. 빌드 타임 코어 vs 런타임 플러그인 모델 충돌

**분류**: 모순

코어 설계: 컬렉션/타입/스키마/API/Admin UI는 `nexpress.config.ts`에서 빌드 타임에 생성된다.

플러그인 설계: 플러그인이 필드, 라우트, 컬렉션, 스케줄 태스크를 추가할 수 있다고 명시한다.

이 두 모델은 양립 불가하다. 플러그인이 컬렉션을 추가하면
DB 스키마 생성과 마이그레이션이 필요한데, 런타임에는 불가능하다.

**해결 제안**: v1에서 플러그인 설치 = 코드 변경 + 리빌드/재시작으로 명시.
런타임 플러그인은 hooks/actions만 허용.

### CB-3. 라우팅 모델 미완성

**분류**: 모순

- `[...slug]` catch-all이 모든 경로를 소비하는데, 시스템 경로(`/sitemap.xml`, `/robots.txt`)와의 우선순위 규칙이 없다
- SEO 플러그인이 `/sitemap.xml`을 정의하지만, 플러그인 라우트는 `/api/plugins/{id}/` 아래에 네임스페이스된다. 실제 sitemap은 사이트 루트에 있어야 한다
- 페이지 slug, 블로그 slug, 컬렉션 slug, 관리자 경로, 플러그인 경로 간 충돌 해결 규칙이 없다
- 예약된 경로 목록 (`/admin`, `/api`, `/media` 등)이 정의되지 않았다

### CB-4. 스케줄러/워커 아키텍처 부재

**분류**: 누락

플러그인 시스템이 `NxScheduledTask`와 cron 표현식을 정의하지만, 실제 실행 인프라가 없다.

요청 핸들러에서 실행할 수 없는 작업들:

- 이미지 리사이즈 (sharp 처리가 업로드 요청 안에서 동기 실행 — 대용량 파일 시 타임아웃)
- 스케줄 태스크, webhook 발송, Import/Export 처리, 검색 인덱싱
- 멀티 노드 배포 시 중복 실행 방지

### CB-5. 쓰기 작업 일관성 모델 부재

**분류**: 누락

콘텐츠 저장 시 다음이 원자적으로 이루어져야 하는데, 트랜잭션 경계가 정의되지 않았다:

1. 콘텐츠 row 업데이트
2. `nx_revisions`에 스냅샷 쓰기
3. 관계/조인 테이블 업데이트
4. 캐시 무효화 (`revalidateTag`)
5. 검색 인덱스 갱신
6. webhook/hook 실행

이들이 각각 독립 실행되면 데이터-캐시-인덱스 간 불일치가 발생한다.

### CB-6. 권한 모델이 너무 단순함

**분류**: 누락, 보안

4개 역할(admin/editor/author/viewer)만 있고,
생성된 API/Admin/폼에 대한 컬렉션별, 오퍼레이션별 접근 제어가 불명확하다.

`defineCollection`에 `access` 속성이 있지만,
이것이 API 라우트 핸들러에서 어떻게 enforcement되는지 설계에 없다.

## 🟠 높은 심각도 (High Severity) — 8건

### HS-1. "Zod 빌드 타임 검증"만으로는 부족

**분류**: 누락

설계서에 "Zod validation at build time"이라고 명시되어 있지만,
런타임 검증이 반드시 필요한 경로:

- Admin UI 폼 제출 (API 핸들러)
- REST API 쓰기 (외부 호출)
- Import API
- 플러그인 액션/훅
- 에이전트 API 호출

빌드 타임에 생성된 Zod 스키마를 런타임에도 사용한다는 명시가 필요하다.

### HS-2. 플러그인 격리 약속이 실제 enforcement를 초과

**분류**: 보안

| 약속 | 실제 enforcement |
|---|---|
| 플러그인 CSS는 `@layer nx-blocks`만 사용 | ESLint 경고뿐, 런타임 enforcement 없음 |
| `ctx.next.revalidatePath`는 자기 경로만 | 제한 메커니즘 미설계 |
| 네트워크 접근은 `allowedHosts`만 | Stage 1에서 enforcement 없음 |
| Capability 기반 접근 제어 | Stage 1에서 검증 없음 |

"Trusted plugin"이 빠르게 "프로세스 내 임의 코드 실행"이 되는 현실을 정직하게 문서화해야 한다.

### HS-3. 스키마 진화 정책 미정의

**분류**: 누락

Code-first 생성은 "최초 생성"에만 안전하다. 다음 시나리오가 미정의:

- 필드 이름 변경 / 타입 변경 시 기존 데이터 마이그레이션
- 컬렉션 삭제 시 데이터 처리
- 플러그인 언인스톨 시 플러그인이 생성한 데이터 정리
- 마이그레이션 롤백 전략

### HS-4. 미디어 생명주기 불완전

**분류**: 누락

- 참조 추적 없음: 미디어 삭제 시 해당 미디어를 참조하는 포스트/페이지가 깨짐
- 삭제 의미론 미정의: soft delete? hard delete? 참조 중인 미디어는 삭제 차단?
- 업로드 제한 미정의: 파일 크기 제한, MIME 타입 화이트리스트의 구체적 값
- sharp 처리가 요청 인라인: 대용량 이미지 시 응답 지연

### HS-5. Import/Export 안전성 부족

**분류**: 누락

- 컬렉션 정의를 SKIP하지만, 대상 스키마와의 호환성 사전 검증(preflight) 없음
- 관계형 ID 매핑 (다른 환경의 UUID가 충돌)
- 미디어 바이너리 처리 (참조만? 실제 파일 포함?)
- 부분 실패 시 롤백 또는 idempotent 재시도
- 트랜잭션 경계

### HS-6. 분산 캐시 전략 미정의

**분류**: 확장성

ISR/`revalidateTag`는 단일 노드에서만 유효하다.
셀프 호스팅 멀티 인스턴스 배포 시:

- 한 노드에서 캐시 무효화해도 다른 노드는 stale 데이터 서빙
- Next.js 캐시 핸들러 커스텀 또는 공유 캐시 전략 필요

단일 노드만 지원한다면 명시적으로 문서화 필요.

### HS-7. 검색 구현 부재

**분류**: 누락

Admin UI에 검색 UI(`<Input placeholder="Search...">`)가 있지만,
백엔드 검색 구현이 없다.

- 최소한 PostgreSQL FTS (`tsvector`/`tsquery`) 기반 설계 필요
- 또는 v1에서 검색 UI 제거

### HS-8. 에러 처리 / 관측성 부재

**분류**: 누락

- 표준 API 에러 응답 형식 미정의 (현재 각 핸들러가 ad-hoc `{ error: "..." }` 반환)
- 글로벌 에러 바운더리 (Admin, Site 각각)
- 구조화된 로깅 전략
- 인증/콘텐츠/플러그인 변경에 대한 감사 로그 (audit log)

## 🟡 중간 심각도 (Medium) — 7건

### MS-1. Revision JSONB 스냅샷 DB 비대화

**분류**: 확장성

autosave + 대형 블록 페이지 시 `nx_revisions`가 가장 큰 테이블이 된다.

- retention/pruning 정책 필요
- autosave vs 정식 revision 구분 필요
- 압축 또는 delta 저장 검토

### MS-2. 로컬 미디어 스토리지와 수평 확장 충돌

**분류**: 확장성

`LocalStorageAdapter`는 개발/단일 노드에서만 유효하다.
이 제약을 명시하거나, 프로덕션에서는 S3 필수로 문서화해야 한다.

### MS-3. 테마 JSON → `<style>` 주입 시 CSS injection 위험

**분류**: 보안

관리자가 입력한 테마 값이 그대로 CSS에 삽입된다.
값 화이트리스트/sanitization이 필요하다.

예: `oklch(0.55 0.20 250); } body { display: none` 같은 입력이 가능할 수 있다.

### MS-4. `unstable_cache` 의존

**분류**: 위험

이름 그대로 불안정 API다.
내부 캐시 추상화 레이어로 감싸서 향후 교체 가능하게 해야 한다.

### MS-5. Admin 상태 관리 미정의

**분류**: 누락

v1에서는 App Router + 로컬 폼 상태로 충분하지만,
다음 시나리오에 대한 전략이 필요하다:

- Optimistic update
- Autosave
- 동시 편집 충돌 감지 (optimistic locking / version check)
- Dirty form navigation 경고

### MS-6. Draft/Preview 캐시 누출 위험

**분류**: 보안

Draft Mode와 페이지 캐싱의 엄격한 분리 없이,
미공개 콘텐츠가 공개 캐시에 오염될 수 있다.

- draft 요청은 캐시 bypass 보장 필요
- preview 토큰의 만료/서명 검증 필요

### MS-7. CORS / Rate Limiting / CSP 미정의

**분류**: 보안

API가 외부 에이전트에게 노출되는 설계인데, 기본적인 보호가 없다.

- CORS: 허용 origin 정책
- Rate Limiting: API 엔드포인트별 요청 제한
- CSP: Content Security Policy 기본 헤더

## 🔵 참고 사항 (Not Blockers)

다음은 차단 사항은 아니지만, 설계서에서 의도적으로 제외했는지 확인이 필요한 항목이다.

| 항목 | 비고 |
|---|---|
| Redis / 외부 캐시 레이어 | v1에서 Postgres + Next 캐시로 충분할 수 있음 |
| CI/CD 파이프라인 | 아키텍처 이슈가 아님 |
| i18n (다국어) | 기획서에 "향후"로 명시됨 |
| 실시간 협업 편집 | MVP 제외로 명시됨 |
| 멀티사이트 | MVP 제외로 명시됨 |
| 접근성 (a11y) | Admin UI에 shadcn/ui 사용하므로 기본 수준은 확보되나, 명시적 정책이 없음 |
| 비밀번호 정책 | 8자 최소만 있고, 복잡성 요구사항 없음 |
| 비밀번호 재설정 (이메일) | v1 로컬 관리자용이면 불필요할 수 있음 |

## 주의할 점

### Draft/Preview 캐시 누출
Draft Mode와 페이지 캐싱의 엄격한 분리가 없으면 미공개 콘텐츠가 공개 캐시에 오염된다.

### Plugin Trust 범위 인식
"Trusted plugin"은 "프로세스 내 임의 코드 실행"과 동의어다.
v1 모델이 이것이라면 정직하게 문서화해야 한다.

### Revision/Autosave 데이터 증가
리치 에디터 스냅샷은 예상보다 빠르게 가장 큰 테이블이 된다.

## 에스컬레이션 트리거

다음 요구사항이 추가되면 현재 설계로는 불가하며, 별도 설계가 필요하다:

- **리빌드 없이 서드파티 플러그인 런타임 설치/제거** → 훨씬 제한적인 확장 모델 필요
- **Day 1부터 멀티 노드 셀프 호스팅** → 공유 캐시/작업/스케줄러 시맨틱 필요
- **v1에서 비신뢰 플러그인** → CMS 기능이 아니라 별도 플랫폼 프로젝트로 취급
